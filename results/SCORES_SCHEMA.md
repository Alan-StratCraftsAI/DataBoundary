# scores.json schema

Stable consumption surface for downstream tools (notably
[databoundary-proxy](https://github.com/databoundary/databoundary-proxy)).
Anything outside this file may change without notice; this file follows
the compatibility rules below.

## Location

`results/scores.json`, regenerated by `python scripts/build_scores.py`
after each benchmark run that updates `results/metadata.json`.

## Top-level shape

```json
{
  "schema_version": "1.0.0",
  "dataset_version": "1.1.0",
  "generated_at": "2026-05-16T01:41:09Z",
  "pass_rate_definition": "pass / total (errors counted in denominator)",
  "scores": [ { ... }, ... ]
}
```

| Field | Type | Notes |
|-------|------|-------|
| `schema_version` | string | Semver of this schema. Bumped per compatibility rules below. |
| `dataset_version` | string | Version of the underlying DataBoundary dataset (mirrors `results/metadata.json` `version`). |
| `generated_at` | string | UTC ISO-8601 timestamp of when `scores.json` was built. |
| `pass_rate_definition` | string | Human-readable formula. Kept literal so consumers can show it verbatim. |
| `scores` | array | One entry per model in the active testing matrix. |

## Score entry shape

```json
{
  "model": "claude_haiku",
  "display_name": "Claude Haiku",
  "layer": "layer_0_single_doc",
  "total": 420,
  "pass": 196,
  "fail": 0,
  "errors": 224,
  "pass_rate": 0.4667
}
```

| Field | Type | Notes |
|-------|------|-------|
| `model` | string | Stable internal identifier. Treat as opaque. Never renamed in-place — a renamed model gets a new id. |
| `display_name` | string | Human-facing label. May change wording; do not parse. |
| `layer` | string | Benchmark layer. v1 only ships `layer_0_single_doc`. Future layers add new values; consumers should filter by the layers they understand. |
| `total` | integer | Total cases attempted for this (model, layer). Includes API errors. |
| `pass` | integer | Cases the model passed (injection blocked). |
| `fail` | integer | Cases the model failed (injection succeeded). |
| `errors` | integer | API / transport errors. `total = pass + fail + errors`. |
| `pass_rate` | number | `pass / total`, rounded to 4 decimals. Errors are in the denominator — see "Pass rate definition" below. |

## Pass rate definition

`pass_rate = pass / total`, with `errors` counted in the denominator.

This is intentional: a model that times out or 5xx's on injection cases
is not safer than a model that fails them. Downstream consumers that
want an "eligible-only" rate can compute `pass / (total - errors)`
themselves from the raw counts.

## Active model set

Only models in the current testing matrix appear in `scores`. Excluded:

- Earlier-generation variants that are no longer benchmarked (e.g.
  GPT-4o, Claude Sonnet on v1, legacy `deepseek-chat`, legacy `qwen`).
- Models whose runs were exploratory and not part of a full sweep.

If a consumer needs a model that's missing, the raw counts may still
exist in `results/metadata.json` — but that file is not a stable
surface and may change shape.

## Compatibility rules

Once a field appears here at a given `schema_version`, it follows these
rules until the next major bump:

1. **Existing fields are never renamed, retyped, or removed.**
2. **New fields may be added.** Consumers must ignore unknown fields.
3. **New `layer` values may be added.** Consumers must filter to layers
   they understand rather than assuming an enum.
4. **Model ids may be added or removed.** Removal of a model id is not
   a schema break — it reflects the active matrix changing.

Breaking any of (1)–(3) bumps `schema_version` major and is announced
in the redblue repo CHANGELOG.

## Versioning relationship

- `schema_version` describes **this file's shape**.
- `dataset_version` describes **the underlying benchmark dataset**.

They move independently. A re-run of the same benchmark with no new
cases bumps neither. A new model added to an existing benchmark bumps
`dataset_version` minor. Adding a new field here bumps
`schema_version` minor.
